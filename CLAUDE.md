# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
pip install -e .                          # Dev install
python -m uvicorn web_counter.main:app    # Dev server (foreground, auto-reload)
web-counter start --salt xxx              # Production start (background, PID-managed)
web-counter stop                          # Stop background process
web-counter restart                       # Stop + start
web-counter createsuperuser               # Interactive admin creation
web-counter export-js > counter.js        # Extract frontend JS to file
python -m build                           # Build sdist + wheel
twine upload dist/*                       # Publish to PyPI
```

No test suite exists. Manual verification:
```bash
export COUNTER_SALT="test123"
python -m uvicorn web_counter.main:app &
curl http://localhost:8000/api/health          # → {"status":"ok"}
curl -X POST localhost:8000/api/visit -H 'Content-Type: application/json' -d '{"path":"/test"}'
curl http://localhost:8000/api/count?paths=/test
```

## Architecture

```
config.py        ← standalone (auto-loads .env on import)
models.py        ← standalone
database.py      ← config.py
rate_limit.py    ← config.py (reads rate_limit setting)
auth.py          ← standalone (in-memory sessions, no DB dependency in this module)
dashboard.py     ← generates HTML strings, no templating engine
main.py          ← all above (create_app factory + module-level `app` singleton)
cli.py           ← main.py, config.py, database.py, auth.py
```

### Module responsibilities

- **`config.py`** — `_load_dotenv()` runs at module import time. `Config(**kwargs)` merges kwargs > env > .env > defaults. Must call `.validate()` to enforce `salt` is set.
- **`models.py`** — Pydantic v2 models: `VisitRequest`, `OffsetRequest`, `ResetRequest`, `LoginRequest`.
- **`database.py`** — All async SQLite via aiosqlite. Every function opens a new connection (WAL mode), does its work, commits, and closes. No connection pooling. Tables: `visits`, `offsets`, `admin_users`, `page_titles`.
- **`rate_limit.py`** — `RateLimiter` class with per-IP token bucket, 60s sliding window, stored in a `defaultdict`.
- **`auth.py`** — bcrypt password hashing. Sessions stored in module-level dict (`token → {username, created_at}`), 24h expiry, not persisted to DB.
- **`dashboard.py`** — Two pure functions returning complete HTML documents: `get_login_html(error)` and `get_dashboard_html(today_pv, today_uv, site_pv, site_uv, offsets, daily_stats)`. Dashboard embeds initial data as JSON and uses Chart.js CDN for the 30-day trend chart.
- **`main.py`** — `create_app(config)` factory builds the FastAPI app. Module-level `app = create_app()` singleton reads config from environment. Routes use `_get_user(request)` helper that reads the `session` cookie and calls `validate_session()`.
- **`cli.py`** — `start` spawns uvicorn via `subprocess.Popen`, passes config as env vars, writes PID file. `stop` sends SIGTERM, waits 10s, falls back to SIGKILL, removes PID file.

### Key design details

**Offset system**: Cumulative counts (site_pv, site_uv, per-page) are `actual + offset`. The `offsets` table stores keys like `site_pv`, `site_uv`, `page_/blog`. Today metrics are never offset. Offsets are set via dashboard or `POST /api/admin/offset`, and are applied at query time in `get_counts()`.

**Two login paths**: Form-based HTML login goes through `POST /dashboard` (reads `request.form()`); programmatic API login goes through `POST /api/admin/login` (reads JSON body). Both set the same `session` cookie.

**Visitor privacy**: `visitor_hash = SHA256(client_ip + salt)`. Raw IPs are not stored. The salt is set once at deployment — changing it invalidates all existing visitor hashes (UV counts reset to zero).

**Package data**: `counter.js` is bundled in the wheel via `[tool.setuptools.package-data]`. The `templates/` directory exists but is currently empty — dashboard HTML is generated by Python functions in `dashboard.py`, not loaded from template files.

**Frontend counter.js**: Discovers API base URL from `data-counter-api` attribute or `document.currentScript.src`. Scans DOM for `data-pv-today/uv-today/pv-site/uv-site/pv-page` attributes. Concurrently POSTs `/api/visit` and GETs `/api/count`. Counters are initially `display:none` and only revealed on successful data fetch. Supports `data-counter-style` values: `default`, `badge`, `card`, `bordered`. All errors are silently swallowed.

**SPA support (v0.2.0)**: Uses `MutationObserver` with 150ms debounce to detect DOM changes from client-side navigation. When counter elements are re-inserted into the DOM, automatically re-scans and re-fetches counts. Multiple counter containers on a page are all made visible independently.

**Leaderboard (v0.3.0)**: `GET /api/top?limit=N&exclude=PATHS` returns top pages by view count with titles from `page_titles` table. Supports `*` wildcard excludes (`*/index.html`). Dashboard management UI for exclude list and display limit. Frontend widget: `<ol data-pv-top="10"></ol>`. Page titles auto-collected from `document.title` on each visit.

**Integration skills**: `skills/rspress-web-counter/` and `skills/vitepress-web-counter/` provide step-by-step integration guides for each framework, covering head config, footer stats, afterDocContent/doc-footer-before counters, leaderboard, and deployment flows.

**Integration caveats**:
- Rspress `head` config: Do not use boolean attributes (`async: true`). Rspress serializes `{ async: true, src: '...' }` as `<script asyncsrc="...">` (attribute concatenation bug). Use `{ src: '/counter.js' }` without boolean attrs.
- JSX/TSX: `data-pv-page` must be written as `data-pv-page=""` (explicit empty string). In JSX, `<span data-pv-page></span>` renders as `data-pv-page="true"`, causing counter.js to use `"true"` as the page path instead of `window.location.pathname`.
- Rspress `footer.message` only renders on the homepage. For article-only counters (like page views), use `afterDocContent` in the theme layout. See `DEPLOY.md` for full examples.
